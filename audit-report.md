## Executive summary

**Overall health: 7.5 / 10 (Good) · Critical issues: 0**

This is a single-file vanilla-JS Progressive Web App (`index.html`, ~1,340 lines of inline JS) with a 54-line service worker and **zero third-party dependencies** — so there is no supply-chain / CVE surface, and the security primitives are mostly sound (no `eval`/`new Function`/`document.write`, the Anthropic API key is correctly kept out of backups/URLs/logs, the coach's LLM output is escaped before rendering, and the `?hl=` health import only stores allow-listed numeric values). The codebase is well-tested for a hobby project (~338 assertions). The real gaps are narrow and concrete: **two missing `esc()` calls** create stored-XSS sinks (practically exploitable only via a maliciously-crafted *imported backup file*, which could exfiltrate the stored API key); a **workout-render hot path re-scans all of localStorage many times per set tap** (not yet user-perceptible, but scales with history); the **test suite is not committed to the repo** (it lives in `/tmp`); and **all data lives only on one device** with manual-only backup — the single largest real-world risk to the user. None rise to CRITICAL given this is a solo, single-user, no-backend app.

> Threat-model note: in a single-user local app, "self-XSS" (you typing a script into your own field) is not a meaningful attack. The XSS findings matter because of the **backup import** path — a backup `.json` shared/AirDropped by someone else is restored verbatim, and a crafted exercise/test name in it would execute in the app origin and can read `localStorage` (including `coach_key`). If this app ever gains cross-user sync, treat those two findings as CRITICAL.

## Findings (prioritized)

### 🔴 HIGH

**H1 · Security · Stored XSS via custom test name — `index.html:1239`**
`'<div class="st-lbl">'+t.name+'</div>'` renders `t.name` **unescaped**. The name is free-text from the "add custom test" input (`addCustomTest`, 1278) and is also restorable from an imported backup (`custom_tests` is in the backup allow-list). The sibling renderer at 1254 already uses `esc(t.name)`, so this is an inconsistency. A crafted name (`<img src=x onerror=...>`) executes JS in the app origin and can read `localStorage`, including the plaintext Anthropic key.
**Fix:** wrap in `esc(t.name)`. **Effort:** 2 min.

**H2 · Security · Stored XSS via workout exercise name — `index.html:1648`**
`'<div class="wo-ex-name">'+ex.name+'</div>'` renders `ex.name` **unescaped**. In normal use exercise names come from hardcoded templates/library, but full `wo_*` session objects are restored verbatim by `handleImportFile` (1054), so a malicious backup with `exercises[].name = "<img ...>"` injects here. Every other exercise-name print escapes (e.g. 1676 `esc(prs[pi].name)`).
**Fix:** wrap in `esc(ex.name)`. **Effort:** 2 min.

**H3 · Tests · The test suite is not in the repo — `/tmp/smoke.js`**
The only test asset (~338 assertions encoding hard-won behavior: v13 override-healing, reset/undo, health ingestion, bilateral timing) lives in `/tmp`, untracked by git. It can't run in CI and is one container reset away from being lost permanently. `git ls-files` confirms no committed tests and no CI workflow.
**Fix:** commit it as `test/smoke.js`, add a `package.json` with `"test": "node test/smoke.js"`, and a minimal GitHub Actions workflow. **Effort:** ~30 min.

**H4 · Architecture · Data lives only on one device (data-loss exposure)**
All training history is in this browser's `localStorage`. Mitigated only by a backup nag that fires after 14 days (`updateBackupNag`, 1050) and manual export. iOS evicts non-installed-PWA storage under pressure / after ~7 days idle, and "Clear website data" wipes everything. This is the highest *real-world* severity item — the user has already lost a day's workout once this project.
**Fix:** make the nag more aggressive when `hasUserData()` is true and there's never been a backup; consider periodic auto-export via the existing `navigator.share`/File path; opt-in cloud sync later. **Effort:** ~30 min for nag tuning; more for auto-export.

**H5 · Performance · Workout render re-scans all of localStorage many times per set tap — `index.html:1770, 1494, 1636, 1580, 1498`**
`woLogSet` (1770) calls `renderWorkout()` on every set toggle. For each exercise, the render calls `woHistory` (1494), `woExRpeHist` (1636), and (for ready-to-progress lifts) `woStalled` (1580) — each a **full `localStorage` scan that `JSON.parse`s every `wo_` session**. A finished day also runs `woSessionPRs` (1498, one `woHistory` per exercise). A 4-exercise day with N saved sessions is ~8–12 parse-all-history passes **per checkmark tap**; cost grows linearly with accumulated history. Not user-perceptible at today's data sizes, but it scales poorly.
**Fix:** parse `wo_` sessions once into an in-memory index (invalidate on `woSave`) and have `woHistory`/`woExRpeHist`/`woSeed`/`coachSessions` read from it; optionally repaint only the toggled row instead of a full `renderWorkout()`. **Effort:** ~2–3 h (index); ~30 min for a per-render memo alone.

### 🟡 MEDIUM

**M1 · Architecture · No data-version key / no migration gate — `index.html:1044`**
`collectBackup` writes `version:1` but nothing ever reads it, and there's no top-level `data_version` in `localStorage`. Format changes are "healed on read" in scattered places (`woCleanOverrides`/`woTmplValid` silently delete corrupt overrides — the v13 fix — and load-time backfill mutates old sessions). It works, but a future format change risks another v13-class incident with no version to branch on.
**Fix:** add a single `data_version` key set at load, and a `migrate()` run once at startup that centralizes the existing heal calls. Don't rebuild storage — just add the gate. **Effort:** 1–2 h.

**M2 · Code quality · Save failures are swallowed silently — `index.html:901, 1575, 1214` (+ `913/915/919`)**
`save()`, `woSave()`, `saveTestLog()`, and the custom-list savers all do `try{ localStorage.setItem(...) }catch(e){}`. On quota-exceeded or Safari Private mode the write fails silently and the user believes their workout was saved — a data-loss footgun given the local-only model. (The *import* path already counts/reports failures — mirror that.)
**Fix:** on a save `catch`, surface a one-time toast/nag ("couldn't save — storage may be full"). **Effort:** ~1 h.

**M3 · Performance · `weekStats`/`weekTonnage`/`activityBounds`/`mobStreak` re-scan on every week render — `index.html:1019, 1499, 893, 894`**
Each `setView('week')` / week navigation triggers ~12 `loadStateForDow` reads, 7 `wo_` parses, a full `localStorage` scan (`activityBounds`), and up to 400 sequential reads (`mobStreak`, though it short-circuits at the first gap). Per-navigation, not per-keystroke, so bounded — but it can reuse the same session index as H5.
**Fix:** fold into the H5 index; the `mobStreak` 400 cap is acceptable as-is. **Effort:** ~30 min once the index exists.

## Verified clean (checked, no issue found)

- **No dangerous primitives:** no `eval`, `new Function`, `document.write`, `javascript:` URLs, `insertAdjacentHTML`, or `outerHTML` anywhere.
- **Secret handling:** `coach_key` is never in backups (`isBackupKey` excludes it), never logged (no `console.log` in the file), only sent via the `x-api-key` header to `api.anthropic.com`, and rendered only into an escaped `<input type=password value>`.
- **Coach LLM output:** `coachMd()` escapes (`esc`) *before* applying markdown regexes, so model text can't inject HTML; coach edit cards escape every model-supplied string.
- **`?hl=` health import:** only stores allow-listed numeric values; the free-text `date` is parsed to ints for a storage key and never rendered; the URL param is stripped via `history.replaceState`.
- **Service worker:** ignores cross-origin requests, caches only same-origin GET 200/basic responses, network-first for navigations — no cache-poisoning vector. Cache is `ht-v17`.
- **Timers:** rest timer, stretch timer, stopwatch, and keep-alive intervals are all cleared on their exit paths — no leaks found. Rest timer ticks at 250ms but only repaints on whole-second change.
- **Notes textarea:** `oninput` saves only (`woSession.notes=v; woSave()`) — no re-render per keystroke.
- **Day-view toggles:** do targeted DOM updates (`classList.toggle`), not full re-renders.
- **Reset/undo safety net & Keep/Erase re-seed dialog:** well covered by tests.
- **Asset integrity:** every `*.webp` referenced exists on disk and matches the `sw.js` cache list.
- **No other dead code:** every other `function`/`var` name has ≥2 references; only `woPrevTxt` and `coachApplyEdit` are unused.
